# Illustrative Walkthrough (Generic Example — Not a Real Incident)

> The following is a fictional, illustrative example constructed to teach the two brownfield patterns, the secrets-isolation apply-order gotcha, and the two-automation-identities gotcha in one continuous story. No project names, PR numbers, or dates below refer to any real client or incident. Use it to understand the *shape* of a remediation, then follow the real client-specific example (if one exists) linked from that client's `references/<client>-config.md`.

**Scenario:** Two applications, "Project Foo" (`<EXAMPLE_APP_A>`) and "Project Bar" (`<EXAMPLE_APP_B>`), were provisioned by an engineer running `gcloud` commands and Console clicks directly against the org — creating the projects, enabling APIs, and binding IAM by hand, under time pressure. No YAML existed for either project in `<FOUNDATIONS_REPO>`; Terraform had never heard of either project's existence. This is drift by definition. A subsequent infrastructure review caught it and required both to be brought under FAST management before further changes could land.

**Remediation, illustrative PR sequence:**

- **PR 1** — added the `<EXAMPLE_APP_A>` project-factory YAML as a starting point (no imports yet, just the desired-state declaration).
- **PR 2** — two different fixes landed together: **Project Foo** got a genuine `terraform import` (Pattern A, `<EXAMPLE_APP_A>-imports.tf`, with the `parent:` folder corrected to match where the project actually lived — the first-pass YAML had assumed a `prod` sub-folder, but the live project was actually parented at the org root); **Project Bar** got a **parallel replacement** project, `<EXAMPLE_APP_B>-2` (Pattern B), because the original project ID was already in use elsewhere and not cleanly reusable — the original Project Bar project was left unmanaged, with decommissioning tracked as an explicit open item.
- **PR 3** — created `<EXAMPLE_APP_B>-sec-core` under the security folder (Google-managed encryption, no KMS, since this client did not require CMEK), granting cross-project `secretAccessor` to the runtime SAs and additive `secretmanager.admin` to `<EXAMPLE_APP_B>-cicd-sa`.
- **PR 4** — declared the empty secret containers (e.g. `DATABASE_URL-*`, `AUTH_*`) in `<EXAMPLE_APP_B>-secrets.yaml` inside the new sec-core project, applied *before* the app-level Terraform wrote any secret versions.
- **PR 5** — granted `<ORG_PREFIX>-iac-tf-sa` (the Terraform-apply automation service account used by Project Bar's own app-level CI/CD pipeline — distinct from the app's release SA below) an `iam_project_roles` entry on the new project, because that automation account had zero permissions there and its `terraform apply` was failing outright trying to create supporting resources (Artifact Registry repo, database instance, monitoring). **This is the "two automation identities" pattern worth internalising:** the account that runs `terraform apply` for an app's own IaC and the account that runs the app's release/deploy pipeline are two different service accounts, and **both** need their roles declared in FAST — never granted ad hoc to unblock a failing apply.
- **PR 6** — added required networking APIs (e.g. `servicenetworking.googleapis.com` / `sqladmin.googleapis.com` for a managed database's private service access) to the project's `services:` list instead of `gcloud services enable`.
- **PR 7** — granted a specific serverless-robot service-agent identity `compute.networkUser` on the specific subnet it needed, declared in the networking stage YAML.
- **PR 8** — granted the app's *release* SA, `<EXAMPLE_APP_B>-cicd-sa`, `roles/iam.serviceAccountUser` (`actAs`) on the runtime SAs and on itself, needed for the deploy pipeline to deploy revisions as those SAs — the release-pipeline counterpart to the IaC-pipeline fix in PR 5.
- **PR 9** — declared a third-party API key secret container in the same sec-core secrets dataset, after a deployment failed to start because it referenced a secret whose container had not been declared yet — reinforcing the apply-order rule in the secrets section of SKILL.md.
- **PR 10** — added an access-binding entry to allow the intended public/invoker access under the org's domain-restricted-sharing policy, by binding a pre-existing conditional tag rather than editing the org policy directly.

Every one of these was a YAML edit plus `terraform plan`/`apply` plus PR. None of them was a `gcloud` write or a Console click. That is the standard this skill enforces on every future GCP change, not just recovery from an incident.
